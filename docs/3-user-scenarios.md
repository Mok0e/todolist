## 테스트 환경 주의사항

- **기술 환경**: 프론트엔드 React 19 + TypeScript, 백엔드 Node.js + Express (JavaScript), DB PostgreSQL 17. 시나리오 검증 시 이 환경을 기준으로 한다.
- **날짜 고정 (Mock)**: 날짜 기반 상태 계산(NOT_STARTED, IN_PROGRESS, OVERDUE)이 포함된 시나리오(US-05, US-07, US-09, US-11)는 테스트 실행 시 시스템 날짜를 시나리오에 명시된 날짜(예: 2026-05-27)로 고정(mock)해야 한다. 백엔드: 서버 시각 주입(의존성 주입 또는 환경변수), 프론트엔드: `Date` 객체 모킹.
- **Optimistic Update 검증**: US-15(테마 전환) 시나리오의 Optimistic Update 검증 기준은 "API 요청 발송 전 UI 상태 전환 완료 확인"이다. 롤백 트리거 조건은 "HTTP 4xx 또는 5xx 응답 수신 시"로 한정한다.

## 전체 흐름 (Overall Flow)

### 1. 사용자 여정 개요

TodoList 앱의 전체 사용자 여정은 **인증 → 준비(카테고리·설정) → 핵심 반복 루프(할 일 관리)** 3단계로 구성된다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                        TodoList 사용자 여정                          │
└─────────────────────────────────────────────────────────────────────┘

[첫 방문]                [재방문]
    │                       │
    ▼                       ▼
┌────────┐           ┌────────────┐
│ US-01  │           │   US-02    │
│ 회원가입│           │   로그인   │
└───┬────┘           └─────┬──────┘
    │  자동: 기본 카테고리  │
    │  생성 (R-CAT-02)     │
    └──────────┬───────────┘
               │
               ▼
    ┌──────────────────────┐
    │      [준비 단계]      │
    │  카테고리 생성/수정   │ ◄── US-13 (생성), US-18 (수정)
    │  설정 (테마·언어)     │ ◄── US-15 (테마), US-16 (언어)
    └──────────┬───────────┘
               │
               ▼
    ┌──────────────────────────────────────────────────┐
    │               [핵심 반복 루프]                    │
    │                                                  │
    │  할 일 등록 ──► 목록 조회 / 필터링               │
    │  US-05, US-06    US-10, US-11, US-12             │
    │       │                                          │
    │       ▼                                          │
    │  할 일 수정 ◄──── 상태 변화                      │
    │  US-07            (자동 계산)                    │
    │       │           NOT_STARTED                    │
    │       │           IN_PROGRESS                    │
    │       │           OVERDUE                        │
    │       ▼               │                          │
    │  완료 처리 ────────────┘                         │
    │  US-09 (DONE)                                    │
    │       │                                          │
    │       ▼                                          │
    │  할 일 삭제                                      │
    │  US-08                                           │
    └──────────────────────────────────────────────────┘
               │
               ▼
    ┌──────────────────────┐
    │      [세션 종료]      │
    │  로그아웃  US-17      │
    │  회원 탈퇴 US-04      │
    └──────────────────────┘
```

### 2. 핵심 흐름별 시나리오 매핑

#### 흐름 A: 신규 사용자 온보딩

> 앱을 처음 사용하는 사용자가 할 일을 등록하기까지의 최단 경로

| 순서 | 시나리오                         | 핵심 행위                 | 결과                                  |
| ---- | -------------------------------- | ------------------------- | ------------------------------------- |
| 1    | **US-01** 회원가입               | 이름·이메일·비밀번호 입력 | 계정 생성 + `기본` 카테고리 자동 생성 |
| 2    | **US-02** 로그인                 | 이메일·비밀번호 입력      | JWT 발급, 목록 화면 진입              |
| 3    | **US-13** 카테고리 생성          | (선택) 전용 카테고리 추가 | 카테고리 준비 완료                    |
| 4    | **US-05** / **US-06** 할 일 등록 | 제목·날짜·카테고리 입력   | 할 일 생성, 상태 자동 계산            |
| 5    | **US-10** 목록 조회              | 카테고리 필터 활용        | 등록한 할 일 확인                     |

---

#### 흐름 B: 일상 업무 관리 루프 (재방문 사용자)

> 매일 반복되는 할 일 처리 사이클

| 순서 | 시나리오               | 핵심 행위                  | 결과                        |
| ---- | ---------------------- | -------------------------- | --------------------------- |
| 1    | **US-02** 로그인       | 앱 접속                    | 테마·언어 설정 자동 복원    |
| 2    | **US-11** OVERDUE 조회 | 기한 초과 항목 파악        | 우선순위 재정립             |
| 3    | **US-07** 할 일 수정   | 마감일 연장 또는 내용 수정 | 변경 내용 저장, 상태 재계산 |
| 4    | **US-09** 완료 처리    | 처리된 할 일 체크          | 상태 → DONE                 |
| 5    | **US-12** 복합 필터    | 특정 프로젝트 × 상태 확인  | 필터된 목록 조회            |
| 6    | **US-17** 로그아웃     | 세션 종료                  | JWT 삭제                    |

---

#### 흐름 C: 학기/프로젝트 마감 정리

> 주기가 끝날 때 데이터 정리 흐름

| 순서 | 시나리오                   | 핵심 행위                       | 결과                                     |
| ---- | -------------------------- | ------------------------------- | ---------------------------------------- |
| 1    | **US-08** 완료 항목 삭제   | DONE 할 일 제거                 | 목록 정리                                |
| 2    | **US-14** 카테고리 삭제    | 더 이상 필요 없는 카테고리 제거 | 소속 할 일 → `기본` 카테고리로 자동 이동 |
| 3    | **US-04** (선택) 회원 탈퇴 | 앱 완전 이탈                    | 계정 + 전체 데이터 cascade delete        |

### 3. 할 일 상태 전이 흐름

> 할 일 상태는 DB에 저장하지 않고 **API 호출 시점의 서버 UTC 기준으로 런타임 계산**된다. (R-STA-01~04)

```
                    [할 일 생성]
                         │
                         ▼
              ┌─────────────────────┐
              │   NOT_STARTED       │  시작일이 미래이거나 날짜 없음
              │   (시작 전)          │  (R-STA-01)
              └──────────┬──────────┘
                         │ 시작일 도래 (오늘 >= start_date)
                         ▼
              ┌─────────────────────┐
              │   IN_PROGRESS       │  시작일 <= 오늘 <= 종료일
              │   (진행 중)          │  (R-STA-02)
              └─────┬───────────────┘
                    │                │
     오늘 > end_date│                │ 사용자가 완료 처리 (US-09)
                    ▼                ▼
         ┌──────────────┐   ┌──────────────────┐
         │   OVERDUE    │   │      DONE        │  (R-STA-03)
         │  (기한 초과)  │   │    (완료)         │
         └──────┬───────┘   └────────┬─────────┘
                │                    │
                │ 사용자가 완료 처리   │ 완료 취소 (US-09)
                ▼                    │
         ┌──────────────┐            │
         │    DONE      │            ▼
         │   (완료)      │   [상태 재계산 → IN_PROGRESS 또는 OVERDUE]
         └──────────────┘   (날짜 조건으로 결정)

  ※ 종료일 당일(오늘 == end_date): IN_PROGRESS (AC-07-09)
  ※ 오늘 > end_date: OVERDUE (AC-07-09a)
```

## 그룹 1: 인증 (Authentication)

### US-01: 신규 사용자 회원가입

**사용자**: 공통
**관련 UC**: UC-01
**사전 조건**: 사용자가 앱에 처음 방문한 상태이며, 아직 계정이 없다.

**시나리오 흐름**:

1. 사용자가 로그인 화면에서 "회원가입" 링크를 클릭하여 회원가입 화면으로 이동한다.
2. 이름 필드에 자신의 이름(예: "김지수")을 입력한다.
3. 이메일 필드에 사용하지 않은 이메일 주소(예: "jisoo@example.com")를 입력한다.
4. 비밀번호 필드에 영문자와 숫자를 포함한 8자 이상의 비밀번호(예: "Study2026")를 입력한다. (R-AUTH-06)
5. "회원가입" 버튼을 클릭하여 제출한다.
6. 시스템은 계정을 생성하고, 동시에 `기본` 카테고리를 자동 생성한다. (R-CAT-02)
7. 회원가입 성공 후 로그인 화면으로 이동한다. (AC-01-01)

**예외 흐름**:

- **중복 이메일**: 이미 가입된 이메일을 입력하고 제출하면 "이미 사용 중인 이메일입니다" 오류 메시지가 표시되며, 가입이 진행되지 않는다. (AC-01-02, R-AUTH-04)
- **비밀번호 규칙 미달**: 숫자가 없는 8자 비밀번호("abcdefgh")를 입력하면 "비밀번호는 8자 이상 128자 이하, 영문자와 숫자를 각각 1자 이상 포함해야 합니다" 오류가 표시된다. (AC-01-03)

**결과**: 계정 생성 완료. `기본` 카테고리가 자동 생성되며, 사용자는 로그인 화면으로 이동하여 앱 사용을 시작할 수 있다. 기본 테마는 `LIGHT`, 기본 언어는 `ko`로 설정된다. (R-SET-02, R-SET-04)

---

### US-02: 기존 사용자 로그인

**사용자**: 공통
**관련 UC**: UC-02
**사전 조건**: 사용자가 이미 계정을 보유하고 있으며, 로그인 화면에 접근한 상태이다.

**시나리오 흐름**:

1. 직장인 사용자 박민준이 출근 후 업무 시작 전 앱에 접속한다.
2. 로그인 화면에서 가입한 이메일 주소를 이메일 필드에 입력한다.
3. 비밀번호 필드에 본인의 비밀번호를 입력한다.
4. "로그인" 버튼을 클릭한다.
5. 시스템이 자격증명을 검증하고 JWT를 발급한다. (AC-02-01)
6. 사용자는 할 일 목록 화면으로 이동하여 오늘 처리해야 할 업무 목록을 확인한다.

**예외 흐름**:

- **잘못된 비밀번호**: 비밀번호를 틀리게 입력하면 "이메일 또는 비밀번호가 올바르지 않습니다" 오류가 표시된다. 이메일 존재 여부는 별도로 노출하지 않는다. (AC-02-02, AC-02-03)

**결과**: 로그인 성공. JWT가 발급되어 인증 상태가 유지되며, 할 일 목록 화면에서 본인의 할 일 전체를 확인할 수 있다. DB에 저장된 테마 및 언어 설정이 즉시 적용된다. (R-SET-01, R-SET-03)

---

### US-03: 프로필 정보 수정

**사용자**: 직장인
**관련 UC**: UC-03
**사전 조건**: 사용자가 로그인한 상태이며, 프로필 화면에 접근해 있다.

**시나리오 흐름**:

**① 이름 변경**

1. 직장인 사용자 이수현이 결혼으로 성이 바뀌어 앱에 등록된 이름을 변경하려 한다.
2. 상단 메뉴 또는 내비게이션에서 "프로필" 화면으로 이동한다.
3. 이름 필드에 새 이름(예: "박수현")을 입력한다. 이메일 필드는 비활성화(read-only) 상태여서 수정할 수 없다. (AC-03-03, R-AUTH-03)
4. "저장" 버튼을 클릭한다.
5. 시스템이 이름을 업데이트하고 성공 메시지를 표시한다. (AC-03-01)

**② 비밀번호 변경** 6. 추가로, 보안을 위해 비밀번호도 변경하려 한다. 현재 비밀번호 필드에 기존 비밀번호를 입력하고, 새 비밀번호 필드에 새 비밀번호(영문+숫자 포함 8자 이상)를 입력한 뒤 저장한다. (AC-03-02)

**예외 흐름**:

- **현재 비밀번호 불일치**: 현재 비밀번호를 잘못 입력하면 "현재 비밀번호가 올바르지 않습니다" 오류가 표시되며, 비밀번호 변경이 이루어지지 않는다. (AC-03-05)
- **새 비밀번호 규칙 미달**: 새 비밀번호가 8자 미만이거나 영문·숫자 중 하나를 포함하지 않으면 규칙 위반 오류가 표시된다. (AC-03-04, R-AUTH-06)

**결과**: 이름 및 비밀번호 변경 완료. 이메일은 변경할 수 없으며, 변경된 이름은 앱 UI에 즉시 반영된다. (R-AUTH-03)

### US-04: 회원 탈퇴

**사용자**: 공통
**관련 UC**: UC-13
**사전 조건**: 사용자가 로그인한 상태이며, 프로필 화면에 접근해 있다. 계정에는 여러 카테고리와 할 일 데이터가 존재한다.

**시나리오 흐름**:

1. 학생 사용자 최은지가 학기가 끝난 후 앱을 더 이상 사용하지 않기로 결정한다.
2. 프로필 화면 하단의 "회원 탈퇴" 버튼을 클릭한다.
3. 시스템이 탈퇴 확인 다이얼로그를 표시한다. "모든 데이터(카테고리, 할 일)가 영구적으로 삭제됩니다. 이 작업은 되돌릴 수 없습니다."와 같은 경고 문구가 표시된다.
4. 사용자가 확인 버튼을 클릭한다.
5. 시스템이 해당 계정과 연관된 모든 카테고리, 할 일 데이터를 cascade delete 처리한다. (R-AUTH-05)
6. 로그인 화면으로 이동한다. (AC-13-01)

**예외 흐름**:

- **탈퇴 후 로그인 시도**: 탈퇴한 이메일로 다시 로그인을 시도하면 자격증명 오류 응답이 반환된다. 계정 존재 여부는 노출하지 않는다. (AC-13-02)
- **탈퇴 후 기존 토큰으로 API 접근**: 탈퇴 처리 후 기존 Access Token으로 API를 요청하면 서버는 해당 user_id 기반 요청을 거부하여 401 Unauthorized를 반환한다. (AC-13-03)

**결과**: 계정 및 모든 데이터(카테고리, 할 일)가 영구 삭제된다. 복구가 불가능하므로 탈퇴 확인 단계에서 충분한 경고가 제공된다.

7. 확인 후 필터 초기화 버튼을 눌러 전체 목록으로 복귀한다.

**예외 흐름**:

- **조건에 맞는 항목이 없는 경우**: 선택한 카테고리에 OVERDUE 항목이 없으면 빈 목록이 표시되며, "해당 조건에 맞는 할 일이 없습니다"와 같은 안내 문구가 표시된다.

**결과**: 특정 프로젝트 범위 내 기한 초과 항목만 정확히 파악하여, 프로젝트별 위험 요소를 효율적으로 관리할 수 있다.

## 그룹 4: 카테고리 관리 (Category Management)

### US-13: 프로젝트별 카테고리 생성

**사용자**: 직장인
**관련 UC**: UC-10
**사전 조건**: 사용자가 로그인한 상태이며, 현재 `기본` 카테고리만 존재한다.

**시나리오 흐름**:

1. 직장인 사용자 이수현이 새 프로젝트를 맡게 되어 해당 프로젝트 전용 카테고리를 만들려 한다.
2. 카테고리 관리 화면(또는 모달)으로 이동한다.
3. 카테고리명 입력 필드에 "클라이언트C 프로젝트"를 입력한다.
4. "추가" 버튼을 클릭한다.
5. 시스템이 이름 유효성(1~30자, 중복 없음)을 검사한다. (R-CAT-05)
6. 검사를 통과하면 카테고리가 생성되고 목록에 표시된다. (AC-10-01)

**예외 흐름**:

- **중복 이름**: "클라이언트C 프로젝트"와 동일한 이름(또는 대소문자만 다른 "클라이언트c 프로젝트")을 다시 생성하려 하면 "이미 존재하는 카테고리 이름입니다" 오류가 표시된다. (AC-10-02, R-CAT-05)
- **이름 길이 초과**: 31자 이상의 이름을 입력하면 "카테고리명은 30자 이하여야 합니다" 오류가 표시된다. (AC-10-03)

**결과**: 신규 카테고리가 생성되어, 이후 할 일 등록 시 해당 카테고리를 선택할 수 있다. (R-CAT-03)

### US-14: 카테고리 삭제 시 할 일 자동 이동

**사용자**: 학생
**관련 UC**: UC-12
**관련 규칙**: R-CAT-04
**사전 조건**: 사용자가 로그인한 상태이며, "1학기 수업" 카테고리가 존재하고 해당 카테고리에 할 일 5개가 포함되어 있다.

**시나리오 흐름**:

1. 학생 사용자 최은지가 1학기가 종료되어 "1학기 수업" 카테고리가 더 이상 필요하지 않다.
2. 카테고리 관리 화면에서 "1학기 수업" 카테고리 옆의 삭제 버튼을 클릭한다.
3. 시스템이 삭제 확인을 요청하며, "해당 카테고리의 할 일 5개가 '기본' 카테고리로 이동됩니다"와 같은 안내 문구를 표시한다.
4. 사용자가 확인을 클릭한다.
5. 시스템이 "1학기 수업" 카테고리를 삭제하고, 해당 카테고리에 속했던 할 일 5개를 자동으로 `기본` 카테고리로 이동시킨다. (R-CAT-04, AC-12-01)
6. 할 일 목록에서 `기본` 카테고리로 이동된 5개 항목을 확인할 수 있다.

**예외 흐름**:

- **기본 카테고리 삭제 시도**: `기본` 카테고리 옆의 삭제 버튼을 클릭하면 삭제가 불가능하다는 오류가 반환된다. `기본` 카테고리는 삭제할 수 없다. (AC-12-02, R-CAT-02)

**결과**: 사용자 정의 카테고리가 삭제되어도, 소속 할 일은 유실되지 않고 `기본` 카테고리로 안전하게 이동된다. 할 일 데이터 무결성이 유지된다. (R-CAT-04)

### US-18: 카테고리명 수정

**사용자**: 직장인
**관련 UC**: UC-11
**사전 조건**: 사용자가 로그인한 상태이며, "클라이언트C 프로젝트" 카테고리가 존재한다.

**시나리오 흐름**:

1. 직장인 사용자 이수현이 프로젝트명 변경으로 카테고리명을 수정하려 한다.
2. 카테고리 관리 화면에서 "클라이언트C 프로젝트" 카테고리의 수정 버튼을 클릭한다.
3. 이름 입력 필드에 "클라이언트D 프로젝트"를 입력한다.
4. "저장" 버튼을 클릭한다.
5. 시스템이 이름 유효성(1~30자, 중복 없음)을 검사한 후 변경된 이름으로 업데이트한다. (AC-11-01, R-CAT-05)
6. 카테고리 목록에 변경된 이름이 반영된다.

**예외 흐름**:

- **기본 카테고리 수정 시도**: `기본` 카테고리의 수정 버튼을 클릭하면 수정 불가 오류가 반환된다. (AC-11-02, R-CAT-02)
- **중복 이름으로 변경 시도**: 이미 존재하는 카테고리명(대소문자 구분 없음)으로 변경하려 하면 "이미 존재하는 카테고리 이름입니다" 오류가 표시된다. (AC-11-03, R-CAT-05)
- **타인 카테고리 수정 시도**: 다른 사용자의 카테고리 ID로 API를 직접 호출하면 403 Forbidden이 반환된다. (AC-11-04)

**결과**: 카테고리명이 변경되며, 해당 카테고리에 속한 할 일들은 영향을 받지 않고 변경된 카테고리명으로 표시된다.

## 그룹 5: 설정 (Settings)

### US-15: 다크 모드로 전환하고 재로그인 후에도 유지 확인

> **v2 기능**

**사용자**: 공통
**관련 UC**: UC-15
**사전 조건**: 사용자가 로그인한 상태이며, 현재 테마는 `LIGHT`(기본값)이다.

**시나리오 흐름**:

1. 직장인 사용자 박민준이 야간 작업 시 눈의 피로를 줄이기 위해 Dark Mode로 전환하려 한다.
2. 설정 화면으로 이동하여 테마 선택 영역에서 "Dark Mode"를 선택한다.
3. 저장 완료를 기다리지 않고 앱 전체 UI가 즉시 Dark Mode로 전환된다. (AC-15-01, Optimistic Update: 서버 응답 전에 화면을 먼저 바꾸고, 저장 실패 시 되돌리는 방식)
4. 시스템이 백그라운드에서 `theme = DARK` 값을 DB에 저장한다. (R-SET-01)
5. 박민준이 로그아웃 버튼을 클릭하여 세션을 종료한다.
6. 이튿날 아침 다시 로그인한다.
7. 로그인 직후 시스템이 DB에서 `theme = DARK`를 읽어 Dark Mode가 즉시 적용된 상태로 할 일 목록 화면이 표시된다. (AC-15-03, R-SET-01)

**예외 흐름**:

- **저장 실패 시 롤백**: Dark Mode로 전환 후 서버에서 HTTP 4xx/5xx 응답이 반환되면(네트워크 오류 포함), UI가 이전 테마(`LIGHT`)로 자동 복원된다. 사용자는 화면이 잠시 어두워졌다가 밝아지는 것을 경험할 수 있다. (AC-15-06)

**결과**: 테마 설정이 DB에 영구 저장되어 재로그인 후에도 유지된다. 즉시 전환(Optimistic Update)으로 빠른 응답성을 제공하며, 저장 실패 시 안전하게 이전 상태로 복원된다. 비인증 화면(로그인, 회원가입)은 로그인 상태와 무관하게 항상 Light Mode로 표시된다. (AC-15-05)

### US-16: 앱 언어를 영어로 변경

> **v2 기능**

**사용자**: 공통
**관련 UC**: UC-14
**사전 조건**: 사용자가 로그인한 상태이며, 현재 언어는 `ko`(기본값)이다.

**시나리오 흐름**:

1. 외국계 회사에 재직 중인 직장인 사용자 이수현이 영어로 앱을 사용하고 싶다.
2. 설정 화면으로 이동하여 언어 선택 영역에서 "English (en)"을 선택 후 저장한다.
3. 앱 전체 UI가 영어로 전환되며, `language = en` 값이 DB에 저장된다. (AC-14-01, R-SET-03)
4. 이수현이 로그아웃 후 재로그인한다.
5. 로그인 직후 시스템이 DB에서 `language = en`을 읽어 영어 UI가 유지된다. (AC-14-03, R-SET-03)

**예외 흐름**:

- **비인증 화면의 언어 처리**: 이수현이 로그아웃된 상태에서 로그인 화면에 접근할 때, 브라우저 언어가 `en-US`이면 언어 코드 앞 두 자리(`en`)를 기준으로 영어로 표시된다. (AC-14-05, R-SET-05)
- **미지원 브라우저 언어**: 브라우저 언어가 `ja`(일본어)인 경우, 지원하지 않는 언어이므로 한국어(`ko`)를 기본으로 표시한다. (AC-14-06, R-SET-05)

**결과**: 언어 설정이 DB에 저장되어 재로그인 후에도 영어 UI가 유지된다. 비인증 화면은 브라우저 언어를 따르되 미지원 언어는 한국어로 폴백 처리된다. (R-SET-05)

## 그룹 6: 보안 (Security)

### US-19: 타인 데이터 접근 차단 확인

**사용자**: 공통 (보안 테스트 관점)
**관련 UC**: UC-05 (주), UC-06, UC-12
**사전 조건**: 사용자 A와 사용자 B가 각각 계정을 보유한다. 사용자 A가 로그인하여 Access Token을 취득한 상태이며, 사용자 B 소유의 할 일 ID(`todo_id_B`)와 카테고리 ID(`cat_id_B`)를 외부 경로(예: URL 추측, 다른 사용자로부터 전달받음)로 알고 있다.

**시나리오 흐름**:

**① 타인 할 일 수정 시도 (UC-05)**
1. 사용자 A가 curl 또는 API 클라이언트로 `PATCH /todos/todo_id_B` 엔드포인트를 사용자 A의 Access Token을 담아 직접 호출한다.
2. 서버가 토큰을 검증하여 사용자 A의 user_id를 추출하고, `todo_id_B`의 소유자(사용자 B)와 비교한다.
3. 소유자가 일치하지 않으므로 서버는 **403 Forbidden**을 반환한다. 수정은 이루어지지 않는다. (AC-05-03)

**② 타인 할 일 삭제 시도 (UC-06)**
4. 사용자 A가 `DELETE /todos/todo_id_B`를 동일한 Access Token으로 호출한다.
5. 서버가 동일한 소유자 검사를 수행하고 **403 Forbidden**을 반환한다. 삭제는 이루어지지 않는다. (AC-06-03)

**③ 타인 카테고리 삭제 시도 (UC-12)**
6. 사용자 A가 `DELETE /categories/cat_id_B`를 호출한다.
7. 서버가 소유자 검사 후 **403 Forbidden**을 반환한다. 삭제는 이루어지지 않는다. (AC-12-04)

**예외 흐름**:

- **토큰 없이 접근 시도**: Authorization Header 없이 API를 호출하면 소유자 검사 이전 단계에서 **401 Unauthorized**가 반환된다. (인증 및 권한 흐름 다이어그램 참조)

**결과**: 유효한 인증 토큰을 보유한 사용자라도 타인 소유 리소스에 대한 수정·삭제는 항상 **403 Forbidden**으로 차단된다. 할 일 및 카테고리 데이터는 소유자만 변경할 수 있다. (R-TODO-05)
